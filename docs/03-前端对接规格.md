# 03 · MiCall 前端对接规格

> **本文件不是设计文档。前端 UI 已由 `AI Call.dc.html` 原型定稿，不允许改动。** 本文件只定义两件事：① DC 原型 → 生产框架的翻译原则；② 把原型里的 mock 替换成真实服务端对接的清单与信令协议。

---

## 1. 最高原则：原型即真理

- `AI Call.dc.html` 是视觉、交互、文案、动画、状态机的**唯一真相**。
- Claude Code 的任务是把 DC 格式（`<x-dc>` 模板 + 继承 `DCLogic` 的类 + `support.js` 运行时）翻译成生产前端框架（React/Vue），**逐像素、逐文案、逐动画、逐状态对齐**。
- **任何"我觉得可以优化/更合理"的改动都不要做。** 视觉或交互有疑问 → 停下来问，不自作主张。
- 保留的细节包括但不限于：极简主屏（影像 + 单个呼叫键）、分状态精调的动画时长（`orbAnim`/`haloDur`/`fieldDur` 各 phase 不同值）、深浅色模式、字体三档缩放（`fontScale` → `rootZoom`）、所有 sheet（角色/场景/语言/设置/账单/邀请/充值/历史/联系/隐私条款等）、所有文案（如结束态"这次聊得怎么样？"、文字模式结束"明天也别硬撑。"、待机"想聊就轻点呼叫"）。

> 注：原型用"光球"作为角色影像的**占位**。生产版中，光球区域替换为《角色资产生成规范》定义的循环视频影像（固定背景 + 上半身）。这是唯一的视觉替换，其余布局/交互不动。影像区的点击行为保持原型逻辑（`orbTap` → 打开角色详情）。

---

## 2. 状态机：结构不变，触发源改变

原型的状态机 `phase` 流转保持不变，但**驱动方式从"前端定时器"换成"服务端信令"**。

### 原型现状（mock）
- `dial()` 用 `setTimeout(2200ms)` 模拟接通，且 `Math.random()<0.18` 模拟 18% 失败。
- `connect()` 用 `setInterval(1000ms)` 前端自减计时 `remaining`/`seconds`。
- `toListen → toThink → toSpeak` 用 `setTimeout` 循环，台词从 `scenarioDefs[].lines` 轮播。

### 生产版（真实）
- `phase` 枚举不变：`idle → calling → listening → thinking → speaking → ended`，外加**打断态**（`speaking → listening` 硬跳转，跳过 thinking）。
- 状态切换由**服务端控制信令**驱动，前端收到事件后 `setState`，不再用定时器自驱。
- `seconds`/`remaining` 由服务端推送，前端只显示（**计费服务端权威**，见 §4）。

## 3. Mock 替换清单（逐项）

| 原型方法/逻辑 | 现状(mock) | 生产版替换 |
|---|---|---|
| `startCall()` | 检查 mic 权限后 `dial()` | 保留权限检查；权限通过后发起 WebRTC 连接 + 通知服务端建立会话 |
| `grantMic()` | 直接置 `micGranted` | 真实请求 `getUserMedia({echoCancellation:true})`；成功后建连 |
| `dial()` | setTimeout 模拟接通 + 18%随机失败 | 等待服务端"接通"信令；失败由服务端真实结果驱动（**18% 假失败删除**，见下方注） |
| `connect()` | setInterval 前端计时 | 不再前端计时；进入 listening 由服务端信令触发 |
| `toListen/toThink/toSpeak` | setTimeout 轮播假台词 | **全部删除**。listening/thinking/speaking 状态切换 + 字幕全由服务端信令驱动 |
| `seconds` 计时 | 前端自增 | 服务端推送 `remaining`/已用时长，前端显示 |
| `remaining` 自减 | 前端 setInterval | 服务端权威扣减后推送 |
| `endCall()` | 本地切 ended | 通知服务端结束会话 + 切 ended（结束态 UI 不变） |
| `scheduleOpen`（预约）/ `incoming`（来电）相关 | 原型 JS 残留的无 UI 死代码 | **已从前端原型删除**（预约通话 + 来电功能整体移除）。原型中 `incoming`/`scheduleOpen` 状态、`incomingCall`/`answerIncoming` 方法、及 renderVals 里 `incoming`/`incomingName`/`answerIncoming`/`declineIncoming`/`scheduleFromMenu`/`scheduleOpen`/`scheduleClose`/`triggerIncoming` 导出已全部删除。H5 无法可靠做主动来电，不实现。复刻时按删除后的原型即可，无需这些 |
| `confirmSwitch()` | 本地切角色 | 切角色 = 结束当前会话 + 用新 character_id 建新会话 |
| `scenarioDefs[].lines` | 写死台词 | 删除假台词；真实对话来自服务端。场景的 `prompt` 字段保留（作为情境注入，见后端 §3.4） |

> **关于 18% 接通失败**：原型为"拟真"做了随机失败。**生产版删除此假失败**（用户体验官判定：打电话 18% 打不通是 bug 感不是真人感）。接通失败只在真实网络/服务错误时发生，并复用原型的 `callFailed` UI（`retryDial`）。

## 4. 服务端 → 前端 信令协议（控制通道）

通过 WS / WebRTC DataChannel 下行。前端收到即 `setState`。

```jsonc
// 接通
{ "type": "connected" }                         // calling → listening

// 状态切换
{ "type": "state", "phase": "listening" }        // 驱动 phase
{ "type": "state", "phase": "thinking" }
{ "type": "state", "phase": "speaking" }

// 打断（speaking → listening 硬跳转，跳过 thinking）
{ "type": "interrupted" }                        // 前端切 listening + 影像"收住"过渡

// 字幕（实时滚动，对应原型 subtitle / lines）
{ "type": "subtitle", "role": "user", "text": "...", "partial": true }  // 用户说话(ASR partial)
{ "type": "subtitle", "role": "ai", "text": "..." }                     // AI 字幕

// 情绪标签（驱动影像切换；语音 emotion 在服务端已用，前端只需切影像）
{ "type": "emotion", "tag": "tender" }           // 前端 crossfade 到对应循环视频

// 计费（服务端权威）
{ "type": "billing", "remaining_seconds": 700, "elapsed": 12 }

// 低时长/耗尽（复用原型 lowWarned / outOfMins UI）
{ "type": "low_minutes", "remaining_seconds": 60 }
{ "type": "out_of_minutes" }

// 接通失败（真实错误，复用原型 callFailed UI）
{ "type": "call_failed", "reason": "network" }

// 结束
{ "type": "ended" }                              // → ended 态（保留评分/反馈 UI）
```

## 5. 前端 → 服务端 信令（控制通道）

```jsonc
{ "type": "start_call", "character_id": "lin_wan", "scenario": "heart" }  // 发起
{ "type": "end_call" }                                                    // 挂断
{ "type": "mute", "on": true }                                            // 静音（本地也处理麦克风）
{ "type": "switch_character", "character_id": "...", "scenario": "..." }   // 切角色（=结束+新建）
{ "type": "set_scene", "scene": "..." }                                   // 设置/插入场景（情境，非画面）
{ "type": "text_input", "text": "..." }                                   // 文字模式输入（textMode）
```

- 音频上行走 WebRTC 媒体通道（连续 20ms 帧），不走这里。
- `mute`：前端本地停麦克风 + 通知服务端。
- `text_input`：原型有 `textMode`（文字模式），保留；该模式下用户打字，服务端同样返回字幕/语音。

---

## 6. 影像播放（视觉的实时层）

- 影像区播放当前角色的循环视频（来自资产 `loops_url`）。
- 状态/情绪切换：收到 `state` 或 `emotion` 信令 → **用两个 video 元素叠加做 200-300ms crossfade**，不硬切。
- idle 态可在多段 idle 循环间随机穿插（打散周期感）。
- `<video loop muted playsinline>`（H5 必须 `playsinline` 且静音自动播放，音频走 WebRTC 单独通道）。
- 影像与音频**不要求帧级同步**——影像表达状态/情绪氛围，音频是真实语音流，两者解耦播放。

---

## 7. 场景的前端处理（情境，非画面）

- 场景选择/自定义 UI 保留原型（`scenarioOpen`、`customScene`、`customHistory` 等）。
- 用户选/输入场景 → 发 `set_scene` 信令 → 服务端作为情境注入（空间/由头维度），并叠加对应**环境音**。
- **画面不因场景改变**（固定背景），空间感由环境音 + 角色对话承载。
- 模式分野（后端处理为主，前端体现入口）：功能性场景（面试/陪练）vs 关系性陪伴，二者交互入口可有区分，但不在本期强行重构原型。

---

## 8. 不动的部分（明确列出，避免误改）

以下原型功能保持纯前端 + 常规 REST，**不需要实时信令**，照原型实现：
角色列表/详情/收藏/搜索、场景列表/自定义/历史、语言切换、深浅色模式、字体缩放、设置（改密码/取消订阅/隐私条款）、账单、邀请裂变、充值/会员、通话历史、联系我们/工单、登录注册、cookie/隐私弹窗、新手引导。

这些的数据走普通后端 API，与实时通话链路无关。
