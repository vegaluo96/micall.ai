# 02 · MiCall 后端架构与实现规格

> 本文件定义后端的完整架构。组织方式遵循 CLAUDE.md 的"四层防线（木桶定律）"——按"穿帮的时间尺度"分层，每层写到可实现。因项目一次性做完整套，四层必须全部到位。

---

## 0. 总架构图（文字版）

```
客户端(H5)
  ├── WebRTC 媒体通道  ←→  服务端：音频上行/下行
  └── 控制信令通道(WS/DataChannel)  ←→  服务端：状态事件、字幕、情绪标签、计费

服务端会话编排（Pipecat / LiveKit Agents）
  ├── 尺度一 实时管线：VAD → 阿里百炼 Qwen3-ASR-Flash → apiyi LLM → MiniMax TTS → 下行；影像状态切换；打断
  ├── 尺度二 单轮质量：LLM 人格注入 + 情绪标签 piggyback（驱动语音emotion + 影像）
  ├── 尺度三 记忆人格：事实层 + 理解层（Postgres+pgvector）；通话后异步回写
  └── 尺度四 自主演进：角色自主状态层 + 去讨好人格 + 关系演进

离线 worker（不碰实时路径）：理解引擎、时间推进、画像更新（LLM 走 apiyi）

存储：Postgres（结构化）+ pgvector（向量检索）
```

## 1. 尺度一 · 地基：实时语音管线（毫秒级"像活物"）

> 这是木桶的地基。地基塌则全塌。**后端第一个动作是这一层的延迟验证 spike（见 §1.6）。**

### 1.1 传输层：WebRTC + 控制信令双通道

- **媒体走 WebRTC**：音频上行（20ms 帧连续流）、音频下行（TTS 音频）。必须开浏览器 `getUserMedia({echoCancellation:true})`（AEC，防回声误打断的第一道防线）。
- **控制走 WS / DataChannel**：服务端推状态事件、字幕、情绪标签、计费余额；客户端推控制指令。**媒体归媒体，控制归控制。**
- H5 现实约束：iOS Safari 取麦克风必须用户手势触发、后台播放受限。前端"轻点呼叫"即是手势入口。

### 1.2 编排框架：Pipecat 或 LiveKit Agents

不从零自建。用成熟框架封装"WebRTC传输 + VAD + ASR/LLM/TTS流式串接 + 打断处理"。各节点供应商可插拔，对应后台「接口配置」。

### 1.3 会话循环（核心编排逻辑）

每个通话 = 一个常驻协程，持有：上行音频流、`interrupt_event`、可清空的 `tts_queue`。并发跑三个 task：

```
task A — 感知（常驻）:
  for 每 20ms 音频帧:
    feed VAD
    feed 流式ASR → partial 文本，推给前台字幕
    if VAD 检测到用户开口 AND 当前 AI 在 speaking:
        set interrupt_event              # 打断
        清空 tts_queue
        停止下行音频 push
        cancel 当前 LLM task
        前台推 {type:"interrupted"} → 影像/状态切回 listening
    if end-of-turn 判定成立:
        final = ASR 收尾文本
        启动 task B(final)

task B — 思考生成（每轮一次）:
  messages = 组装context（四层记忆 + 角色 + 情境，见 §3、§4）
  buffer = ""
  for token in LLM.stream(messages):     # 走 apiyi（endpoint 可配）
    if interrupt_event.is_set(): break
    buffer += token
    if buffer 命中句末标点(。！？\n):
        tts_queue.put(buffer); buffer=""   # 句子级，首句抢跑
  if buffer: tts_queue.put(buffer)
  记录实际播出的句子 → 用于被打断时的上下文（见 §1.5）

task C — 发声（常驻）:
  while True:
    sentence = tts_queue.get()
    if interrupt_event.is_set(): continue   # 被打断则丢弃
    audio = MiniMax_TTS.stream(sentence, voice_id, emotion)  # 句子级流式
    通过 WebRTC 下行 push audio
    播完一句 → 客户端回 ack（用于追踪实际播出边界）
```

### 1.4 End-of-turn 判定（别只靠静音）

三信号融合，用框架内置的 turn detector（Pipecat SmartTurn / LiveKit turn detector），不自己拼：
- VAD 静音时长（动态阈值）
- ASR 文本语义完整性（未完成则延长静音阈值）
- （可选）韵律语调

参数起点：
```
静音基础阈值（语义完整时）  500ms
静音延长阈值（语义疑未完成） 1200ms
turn detector 置信度        > 0.7
最短发话时长（低于当噪声）   200ms
```

### 1.5 打断（四个时序难点）

1. **回声误打断**：AEC（客户端）+ 服务端兜底（AI speaking 时提高打断 VAD 阈值，要求用户声音持续 >250ms 且能量超阈值才算真打断）。
2. **全链路熔断顺序**：停下行 → 清 tts_queue → cancel TTS → cancel LLM → 处理半截话 → 状态机回 listening。先停下行（用户体感最直接），再向上游逐层 cancel。
3. **防抖窗口**：用户声音持续 >250-300ms 才算真打断；短促"嗯""对"的附和不打断。与 end-of-turn 静音阈值是两套独立参数。
4. **半截话进上下文**：只把**实际播出的句子**记入 assistant 消息（靠 task C 的 ack 追踪边界），标注成 `"……（被打断）"`，让 AI 下一轮能自然接住。绝不把"已生成但没说出口"的部分当作已说。

### 1.6 影像状态切换（视觉的"像活物"）

- 影像 = Coze 生成的循环视频（见《角色资产生成规范》）。状态：idle / listening / speaking / 情绪态。
- 切换由服务端状态事件 + 情绪标签驱动，**粗粒度切换、不要求帧级同步**（这是规避 H5 死穴的关键）。
- 前端用两个 video 元素叠加做 **200-300ms crossfade**，不硬切。
- 嘴型走**路线 A**：表情表达情绪，不做精确口型同步。情绪对了胜过口型准。

### 1.7 延迟红灯与策略（先走 apiyi，卡了再说）

- **TTFT 是延迟主瓶颈。** 长 prompt（四层注入）会推高 TTFT。
- 压制手段：① **prefix caching**（通话内不变的前缀缓存住）；② **句子级首句抢跑**（首句一出即开始 TTS，掩盖剩余生成）；③ **分层注入**（不变的进缓存前缀，每轮变的放后缀）。
- **apiyi 风险**：聚合网关可能 TTFT 抖动、不透传 prefix caching。**策略：先全走 apiyi；endpoint 做成可配（CLAUDE.md 铁律2）；后端第一个动作是延迟 spike——实测 apiyi 长 prompt 的 TTFT 与缓存。** 卡了就把实时对话 LLM 改配置切直连 DeepSeek，对话逻辑不动。

## 2. 尺度二 · 单轮对话质量（秒~分钟"这轮像人"）

### 2.1 情绪标签 piggyback（一处产生、多处消费）

LLM 生成回复时，用结构化输出**顺带**吐一个情绪标签（12选1或更少，与角色 `emotion_map` 的 key 对齐），**不额外调用 LLM**。该标签同时：
- 查 `voice.emotion_map` → MiniMax TTS 的 emotion 参数
- 查 `visual.loops` → 切到对应情绪的循环视频

**一个标签，驱动声音和脸。这是尺度一/二的枢纽。**

实现建议：让 LLM 输出形如 `[emotion:tender] 实际回复文本……` 的前缀，或用 function/JSON 结构化字段。解析出标签后剥离，文本送 TTS、标签送映射表。

### 2.2 情绪追踪（唯一活在实时路径的情境维度）

- 追踪"本通电话此刻的情感温度"，用于：人格安全阀（识别情绪危机时切换关怀）、影像/语气贴合。
- **绝不为它单独调 LLM**。手段：piggyback 在主生成里，或用本地轻量分类器跑 ASR 文本（毫秒级）。

### 2.3 人格注入

system prompt 由四层记忆 + 角色 + 情境组装（§3、§4）。注入"本轮对话策略"（理解引擎产出，见 §3.3），让 LLM 照着演，不用自己现想。

## 3. 尺度三 · 记忆与人格系统（跨对话"同一个连续的人"）

> 这是 MiCall 的护城河。**事实（记录）与理解（推断）必须物理分开存储。**

### 3.1 双层存储

| 层 | 性质 | 存什么 | 存储 |
|---|---|---|---|
| 事实层 | 客观、只增、可检索 | 用户说过的事、对话片段 | Postgres + pgvector |
| 理解层 | 主观、持续修正、有置信度 | 对"这个人是谁"的画像 | Postgres（结构化 JSON） |

豆包能把事实层做满（记住你说过的话），但不会做理解层——推断一个人是谁，对服务几亿人的助手不划算、不安全。**理解层是结构性差异点。**

### 3.2 用户画像 schema（理解层，per-user × per-character）

```jsonc
{
  "user_id": "...", "character_id": "lin_wan",
  "fact_profile": {        // 客观信息（也可由事实层检索补充）
    "job": "", "relations": [], "ongoing": ["在准备面试"]
  },
  "personality_model": [   // "懂"的本体：对这个人的洞察
    { "insight": "焦虑时用自嘲掩饰挫败感", "confidence": 0.7, "evidence": "..." },
    { "insight": "需要的是被听见，不是被建议", "confidence": 0.8, "evidence": "..." }
  ],
  "interaction_prefs": {   // TA 该怎么对待这个用户
    "wants": "被听见 > 被建议", "avoid_topics": [], "style": "给空间，不追问"
  },
  "open_hypotheses": [     // 待验证假设（"变懂的加速度"引擎）
    { "guess": "疑似和家人关系紧张", "confidence": 0.3, "next": "下次留意，轻轻试探" }
  ],
  "relationship": {        // 关系状态（慢演进）
    "stage": "熟络", "last_topic": "面试", "open_threads": ["面试结果"],
    "last_call_at": "...", "shared_refs": ["养的猫叫团子"]
  }
}
```

- 每条 `personality_model` 带**置信度 + 证据**：注入时确定的可表现、猜测的要试探，避免乱下判断的恐怖谷；新证据可修正旧判断。
- `open_hypotheses` 是让画像主动生长的引擎：AI 带着假设进下一次对话去验证/推翻，这是让用户感到"它在越来越懂我"的根源。

### 3.3 理解引擎（离线异步，不碰实时路径）

通话结束触发后台 worker（几秒~几十秒无所谓），用**较强的模型**（走 apiyi，可选更强档），做三件事：
1. **抽取新事实** → 写事实层（含向量化存 pgvector）。
2. **更新画像**：把"本次对话 + 现有画像"喂模型，做**推断和修正**——印证/推翻哪条旧判断、暴露什么新模式、上次的假设是否验证。产出更新画像 + 新假设。
3. **生成"下次怎么懂他"的对话策略**：把抽象画像翻译成具体指令（开场接哪个线头、这次验证哪个假设、哪些话题小心、用什么方式回应）。供下次通话的 §2.3 注入。

### 3.4 Context 组装（每次"该 AI 接话"时）

四层按优先级组装，token 预算下分层裁剪：
- **角色人设**（静态，永远保留）
- **关系状态 + 画像**（慢变，永远保留 —— 它们是人格）
- **情节记忆检索 Top-K**（可伸缩 —— token 紧就少检几条；检索结合语义相似 + 时间衰减 + 情感权重）
- **本轮对话滑窗**（最先被裁）

裁剪优先级写死成规则。不变的部分进 prefix cache 前缀（配合 §1.7 降 TTFT）。

### 3.5 "模糊而温暖"原则（防记忆恐怖谷）

检索出的情节经一层"自然化"再注入，让 AI"大概记得"而非"数据库查询"。宁可"你养的那只猫……叫团子来着？"（模糊+确认），也不要精确复述时间地点。模糊的记忆更像人，也更温暖。

## 4. 尺度四 · 自主状态与演进（长期"独立存在的人"）

### 4.1 角色自主状态层（独立于用户）

每个角色维护一个**和用户画像平级、但完全独立**的自主状态：当前心情、最近"在经历"的事、精力。通话开始时注入，**它不为服务用户存在**——会让 TA 今天可能"有点累/话特别多/有点心不在焉"。

关键：**有时要和用户需求冲突**（用户想倾诉，TA 今天状态也不好："我今天其实也提不起劲，但你说，我听着"）。"TA 不是随时满血待命"是真人感最强来源之一。

### 4.2 离线时间推进（补"挂了电话仍在活着"）

两次通话间隔期，离线任务生成"TA 这段时间的近况"，作为下次通话状态来源。距上次越久，TA 越可能主动提（"这几天我……"）。H5 放弃了主动来电，这块把"TA 有对话之外的生命"补回来。

### 4.3 去讨好人格（人格 prompt 原则 + 安全阀）

- 角色有自己的观点和边界：会不同意、被冒犯有反应、不无脑肯定。结合理解层甚至会**基于对你的理解反驳**（"你又在逞强了"）。
- **安全阀（不可省）**：识别到用户真实情绪危机时，**人格让位于关怀**——平时可以损你、有态度，你真崩溃时立刻认真。何时坚持人格、何时切关怀，写进逻辑，依据 §2.2 情绪追踪。对应 CLAUDE.md 铁律8。

### 4.4 有限性注入（不完美才像人）

- 对话层：会"记不太清"（模糊+确认，见 §3.5）、有不知道的事、会犹豫改口。
- 语音层：用 MiniMax emotion 做停顿、思考音、语气随情绪起伏，不平稳播报。
- **克制全知**：宁可"好像大概记得"，不要精确复述。

## 5. 计费与限流（服务端权威）

- **绝不信前端计时**（前端改 JS 就白嫖）。服务端按实际通话时长扣减 `remaining`，通过控制信令推给前端显示。
- 前端的 `outOfMins` / 低于1分钟提示 等 UI 逻辑全部保留，数字来源换成服务端推送。
- 对应后台「成本与限流」节点。每分钟成本 = ASR(时长) + LLM(token) + TTS(字符) 叠加，需核对会员定价能否 cover。

## 6. 数据模型与服务分层（落到技术栈）

| 模块 | 技术 / 模型（与 Admin 接口配置原型一致） |
|---|---|
| 实时编排 | Pipecat / LiveKit Agents（Python） |
| ASR | 阿里百炼 `qwen3-asr-flash-2026-02-10` 流式（直连，endpoint 可配） |
| 实时对话 LLM | DeepSeek-V4-Flash（先走 apiyi，endpoint 可配，留直连后路） |
| 离线/理解引擎 LLM | Qwen-Long（走 apiyi，延迟无所谓，享灵活） |
| TTS + 音色 | MiniMax TTS 官方 API（直连，流式，emotion，voice_id） |
| 影像 | Seedance（Coze 场外离线生成循环视频） |
| 事实层 + 向量 | Postgres + pgvector |
| 理解层（画像） | Postgres 结构化 JSON（不需向量，整块注入） |
| 离线任务 | 消息队列 + 后台 worker |
| 部署 | 阿里云 ECS |

**角色定义（出厂，全用户共享）与用户关系（per-user，挂 character_id）严格分离**（CLAUDE.md 铁律7）。

### 6.1 三级配置覆盖（与 Admin 原型对齐）

运行时解析配置的优先级链（Admin 原文）：**用户自定义 > 角色配置 > 全局默认**。

- **全局默认**：默认 TTS 引擎/音色、默认记忆深度 Top-K、默认回复长度、全局视觉背景（背景不可被覆盖）。
- **角色配置**：角色 spec 的 `runtime_overrides`（tts_model / realtime_prompt_extra / memory_depth / reply_max_tokens）+ 角色默认 voice_id / 表情库。
- **用户自定义**：per-user × per-character 的自定义音色（对应前端 `genVoicesByChar`/`voiceByChar`、Admin「用户语音」tab）。运行时若用户为该角色设了自定义音色，优先于角色默认音色。

数据模型（user_voice 表）：
```
user_voice: { user_id, character_id, voice_id, label, created_at }
运行时取音色: user_voice(user,char) ?? character.voice.voice_id ?? global.default_voice
```

## 7. 后台 Admin（运营管理端）

对应原型 `MiCall Admin.dc.html`，**完整 11 个 tab，全部实现**：

1. **数据概览**：总用户、今日通话、总通话时长、本月收入等 KPI。
2. **用户管理**：用户列表、分钟数、消费。
3. **角色管理**：角色列表、通话数、被收藏数；点入配置该角色的 voice_id / TTS模型 / **Seedance 表情视频库（每角色 N 种状态 exprCount）** / 实时 prompt / 记忆深度 / 回复长度（角色级覆盖）。
4. **场景管理**：场景列表、使用次数。
5. **通话记录**：时长、详情。
6. **工单反馈**：对应前端联系我们/工单。
7. **订单充值**：订单、金额。
8. **邀请裂变**：邀请列表、奖励。
9. **接口配置**：5 个节点 endpoint + key 可配（与 Admin 原型 `apiSections` 完全一致）：
   - **ASR · 语音识别**（快链路）：`qwen3-asr-flash-2026-02-10`（阿里百炼）。字段：接口地址 / API Key / 识别语言。
   - **LLM · 快脑（通话中）**（快链路）：DeepSeek-V4-Flash（先经 apiyi，可配，卡了切直连）。字段：接口地址 / API Key / 温度 / 回复上限 Token。
   - **TTS · 语音合成**（快链路）：MiniMax TTS（官方直连，emotion）。字段：接口地址 / API Key / 默认 voice_id / 采样率。
   - **LLM · 长记忆脑（通话后）**（慢链路）：Qwen-Long（经 apiyi，离线）。字段：接口地址 / API Key / 最大上下文。即"理解引擎"用的模型。
   - **Embedding · 记忆检索**（慢链路）：向量化 + 检索，存 Postgres + pgvector。字段：接口地址 / API Key / 向量数据库 / 检索 Top-K。

   > 注：影像（Seedance）是**离线资产生成**，不在实时接口配置链路里，由资产管线处理（见《角色资产生成规范》）。LLM 在 Admin 里明确分成"快脑/慢脑"两个独立节点，正是本架构"实时保延迟 / 离线享灵活"的分流落地。支撑"卡了切直连"。
10. **成本与限流**：总链路状态、节点状态、首句响应延迟、每100分钟成本、今日失败率、每小时成本。对应链路监控。
11. **权限管理** + **用户语音**：用户语音 = 用户自定义音色匹配（per-user voice_id），是三级覆盖的"用户自定义"层载体（见 §6.1）。

### 角色资产导入导出（Admin 内，对应原型「角色导入/导出」）
- **导入**：选择资产包/文件 → 跑校验闸门（见《角色资产生成规范》§7）。
- **导出**：导出全部角色完整信息为 JSON（`micall_characters.json`），**供大模型读取后再生成音色与表情资产**（这正是"spec 即本体、资产即编译产物"，与资产规范一致）。

### 新增建议
- **画像调试**：查看 AI 对每个用户建立的理解（personality_model / open_hypotheses），对调产品至关重要。原型未含，建议补。

## 8. 实施顺序（一次性做完，但有内部依赖）

虽然不做 MVP、四层全做，但有技术依赖顺序：
1. **先做尺度一延迟 spike**（验证 apiyi TTFT + H5 实时链路），这是地基可行性的生死验证。
2. 实时管线（尺度一）跑通：含影像切换、打断。
3. 单轮质量（尺度二）：人格注入 + 情绪标签 piggyback。
4. 记忆人格（尺度三）：双层存储 + 理解引擎离线 worker。
5. 自主演进（尺度四）：自主状态 + 去讨好 + 时间推进。
6. 计费、Admin、全链路联调。

资产轨（角色生成）与前端轨可与后端并行推进。
