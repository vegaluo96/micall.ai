# CLAUDE.md — MiCall 项目宪法

> 本文件是 MiCall 项目的最高纲领。所有开发决策以此为准。其余三份规格文档（角色资产、后端架构、前端对接）都服从本文件的原则。每次开发任务开始前，先读本文件。

---

## 0. 一句话定义

MiCall 是一个**让用户感觉在和一个真实的人通话**的移动端（H5）AI 语音陪伴产品。用户打开后接通一个角色，看到 TA 的上半身影像、听到 TA 的声音，和 TA 实时对话。

产品的灵魂不是"语音助手"，也不是"角色扮演内容平台"，而是**一个独立的、会记得你、会随状态变化的"人"**。

---

## 1. 两条第一性原理（不可违背）

### 1.1 前端第一性原理：极简

前端设计已由产品方（设计师）完成并定稿，体现在 `AI Call.dc.html` 原型中。其核心是极致克制：主屏只有一个角色影像 + 一个呼叫键，没有信息流、没有内容瀑布、没有"刷"的入口。

**铁律：前端 UI 不允许改动。** Claude Code 的任务是把 DC 原型 100% 精确复刻成生产代码，逐像素、逐文案、逐动画对齐。任何"我觉得可以优化"的想法都不要执行。视觉或交互有疑问，停下来问，不要自作主张。详见《前端对接规格》。

### 1.2 架构第一性原理：感觉在和真人打电话

后端架构的唯一目标，是让用户产生"在和一个真实的人对话"的稳定幻觉。所有技术决策用一个问题裁决：

> **这个设计，是让"像真人"更真，还是更假？**

凡是让 AI"更有用、更稳定、更全知、更迎合"的设计要警惕——那是把它推回"工具"。真人会有自己的状态、会不完美、不会无条件迎合。

---

## 2. 核心架构模型：四层防线（木桶定律）

"像真人"的判断分布在四个不同时间尺度上。每一层是一道独立的防线，**任一层穿帮，整个"真实的人"的幻觉就崩塌，无法被其他层补偿（木桶定律，不是叠加）**。

| 层 | 时间尺度 | 守什么 | 穿帮信号 | 主要职责模块 |
|---|---|---|---|---|
| 尺度一 | 毫秒级 | 反应像不像活物 | 卡、慢、僵、音画不同步 | 实时语音管线 + 影像切换 |
| 尺度二 | 秒~分钟 | 这一轮反应像不像人 | 答非所问、情绪不对 | LLM 单轮对话质量 |
| 尺度三 | 跨对话 | 是不是同一个连续的人 | "怎么不记得了""怎么变了个人" | 记忆与人格系统 |
| 尺度四 | 长期 | 是不是独立存在的人 | "总顺着我""没有自己" | 自主状态 + 去讨好 + 演进 |

**因为是"一次性做完整套"（无 MVP），四层必须全部实现到位，任何一层留白都会因木桶定律拖垮整体。**

---

## 3. 三轨开发结构

项目切成三条边界清晰的轨道，分别交付：

1. **前端复刻轨**（确定性）：100% 对齐 DC 原型，mock 处换成接服务端信令的占位。可独立先跑（接 mock 后端）。→《前端对接规格》
2. **资产管线轨**（前置地基）：角色资产生成规范 + 后台导入导出校验。是"喂料系统"，先有它才有角色可用。→《角色资产生成规范》
3. **后端四层防线轨**（核心）：四层防线的完整实现。→《后端架构与实现规格》

## 4. 节点选型（最终定稿）

| 能力 | 供应商 / 模型 | 接入方式 | 实时敏感 |
|---|---|---|---|
| ASR 语音识别 | 阿里百炼 `qwen3-asr-flash-2026-02-10` | 直连，流式 | 是 |
| LLM 快脑（通话中） | DeepSeek-V4-Flash（先经 **apiyi**，endpoint 可配） | 可配 | 是 |
| TTS 语音合成 + 音色 | MiniMax 官方 API | 直连，流式 | 是 |
| LLM 慢脑（通话后/理解引擎） | Qwen-Long（经 apiyi，离线） | apiyi | 否 |
| Embedding 记忆检索 | 向量化 + pgvector | 自托管 | 否 |
| 影像生成 | Seedance（Coze 场外离线生成） | 离线 | 否 |
| 记忆存储 | Postgres + pgvector | 自托管（阿里云 ECS） | 否 |

> 节点划分与 Admin 原型 `apiSections` 一致：LLM 明确分"快脑（通话中，DeepSeek-V4-Flash）/ 慢脑（通话后，Qwen-Long）"，这就是"实时保延迟 / 离线享灵活"的分流。先全走 apiyi、endpoint 可配、卡了切直连。

部署环境：阿里云 ECS。

## 5. 全局铁律（所有轨共同遵守）

1. **前端 UI 不许改**：DC 原型是视觉与交互的唯一真相。
2. **所有外部服务 endpoint/key 走配置，不许硬编码**：ASR/LLM/TTS 的接口地址和密钥都从配置读取，对应后台「接口配置」节点。这是"先走 apiyi、卡了再切直连"策略的前提——切换只改配置，对话逻辑不动。
3. **一致性靠"共享锚"在数据层强制，不靠运行时修、不靠人工保证**：角色片段一致靠共享基准帧；角色间背景一致靠全局固定背景；详见资产规范。
4. **情绪标签一处产生、多处消费**：LLM 生成回复时顺带输出情绪标签（piggyback，不额外调用），同时驱动 MiniMax 语音 emotion 和影像状态切换。
5. **实时路径上绝不增加额外的 LLM 往返**：情绪追踪、状态判断等都 piggyback 在主生成里或用轻量本地手段，不为它们单独调 LLM。
6. **记忆要"模糊而温暖"，不要精确复述**：避免数据库式的精确回放（"你3月14日23:07说…"会让用户毛骨悚然）。详见后端规格记忆部分。
7. **角色定义（出厂）与用户关系（运行时生长）严格分离**：角色 spec 是所有用户共享的出厂定义；用户与角色的记忆/关系是 per-user、挂在 character_id 下的独立数据。
8. **用户福祉高于留存指标**：陪伴产品用户常脆弱。识别到情绪危机时，角色人格让位于关怀；不设计利用孤独的成瘾机制。
